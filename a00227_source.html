1 Input Files
2 ===========
3 
4 The following input files are required to run HyPar. They are indicated as being mandatory or
5 optional. If optional, then default values are used if not found.
6 
7 \b Note: It is best to start with the provided examples, understand their input files based on the
8 information on this page, and then modify them to run other cases.

59 
60 \b Notes:
61 + "ndims" \b must be specified \b before "size" and "iproc".
62 + if "input_mode" or "output_mode" are set to "parallel" or "mpi-io",
63  the number of I/O ranks must be specified right after as an integer.
64  For example:
65 
66  begin
67  ...
68  input_mode parallel 4
69  ...
70  end
71 
72  This means that 4 MPI ranks will participate in file I/O (assuming
73  total MPI ranks is more than 4) (see ReadArrayParallel(),
74  WriteArrayParallel(), ReadArrayMPI_IO() ).
75  - The number of I/O ranks specified for "input_mode" and "output_mode"
76  \b must \b be \b same. Otherwise, the value for the one specified last
77  will be used.
78  - The number of I/O ranks must be such that the total number of MPI ranks
79  is an integer multiple. Otherwise, the code will use only 1 I/O rank.
80 + If any of the keywords are not present, the default value is used, except
81  the ones whose default values say "must be specified". Thus, keywords that
82  are not required for a particular simulation may be left out of the
83  solver.inp input file. For example,
84  - a #Euler1D simulation does not need "par_space_type" or "par_space_scheme"
85  because it does not have a parabolic term.
86  - unless a conservation check is required, "conservation_check" can be left
87  out and the code will not check for conservation.
88  - "immersed_body" need not be specified if there are no immersed bodies present.
89  \b NOTE: However, if it is specified, and a file of that filename does not
90  exist, it will result in an error.

92 \section boundary_inp boundary.inp
93 
94 Requirement: \b mandatory
95 
96 Read by: InitializeBoundaries()
97 
98 Description: Specify boundary conditions.
99 
100 Format: ASCII text
101 
102  nb
103  boundary_type spatial_dimension face [extent]
104  boundary_type spatial_dimension face [extent]
105  ...
106  boundary_type spatial_dimension face [extent]
107 
108 where
109 + \b nb is the number of boundaries
110 + This is followed by at least \b nb rows, each row specifying:
111  - \b boundary_type - Type of boundary, eg. #_PERIODIC_, #_EXTRAPOLATE_, #_DIRICHLET_, etc. See include/boundaryconditions.h for a full list.
112  - \b spatial_dimension - The spatial dimension along which this boundary condition applies. Must be in the range 0, 1, ..., #HyPar::ndims-1
113  - \b face - The face on which this the boundary acts (1 means the left face along the dimension, or the low index end, -1 means the right face
114  along the dimension, or the high index end).
115  - [\b extent] - The extent of this boundary (in terms of coordinates, not grid indices). Written as (2 X #HyPar::ndims) real numbers:
116  xmin_0 xmax_0 xmin_1 xmax_1 ... ... xmin_(ndims-1) xmax_(ndims-1), where [xmin_i, xmax_i] define the extent along
117  spatial dimension i. Note that the xmin_i and xmax_i for i = spatial_dimension are ignored, but should be there for the
118  sake of a uniform format (they can be set as 0).
119 + For the following boundary types, an additional line is required right below the line specifying that boundary with inputs specific to that
120  boundary type:
121  - #_DIRICHLET_: the next line should specify the boundary value of the solution at that boundary (the value is assumed to be constant in space
122  and time). Each component of the solution must be specified (i.e. #HyPar::nvars real numbers).
123 
124  u[0] u[1] ... u[#HyPar::nvars-1]
125 
126  - #_SLIP_WALL_, #_NOSLIP_WALL_, #_SW_SLIP_WALL_, or #_SW_NOSLIP_WALL_: the next line should specify the wall velocity (assumed to be constant
127  in space and time). Each component of the velocity must be specified (i.e. #HyPar::ndims real numbers).
128 
129  u[0] u[1] ... u[#HyPar::ndims-1]
130 
131  - #_SUBSONIC_INFLOW_: the next line must specify the inflow density and velocity vector (assumed to be constant in space and time). Each component
132  of the velocity must be specified (i.e. #HyPar::ndims+1 real numbers)
133 
134  rho u[0] u[1] ... u[#HyPar::ndims-1]
135 
136  - #_SUBSONIC_OUTFLOW_: the next line must specify the outflow pressure (assumed to be constant in space and time), i.e., one real number
137 
138  p
139 
140  - #_SUPERSONIC_INFLOW_: the next line must specify the inflow density, velocity, and pressure (assumed to be constant in space and time). Each component
141  of the velocity must be specified (i.e. #HyPar::ndims+2 real numbers)
142 
143  rho u[0] u[1] ... u[#HyPar::ndims-1] p
144 
145  Note that this boundary can also be implemented through the #_DIRICHLET_; however, the flow variables must be converted to the conserved variable
146  form to specify the Dirichlet boundary value:
147 
148  rho rho*u[0] rho*u[1] ... rho*u[#HyPar::ndims-1] E (internal energy)
149 
150  - #_TURBULENT_SUPERSONIC_INFLOW_: the next line must specify the inflow mean density, velocity, and pressure (assumed to be constant in space and time),
151  i.e, #HyPar::ndims+2 real numbersm and the name of the file that contains the unsteady fluctuation data.
152 
153  rho u[0] u[1] ... u[#HyPar::ndims-1] p filename

254 \section petscrc .petscrc
255 
256 Requirement: \b mandatory <B>if using PETSc time integration</B>; if absent, HyPar will use native time integration.
257  If compiled without PETSc, this file is not required.
258 
259 Read by: PETSc
260 
261 Description: This file contains the input flags required by PETSc time integration and its associated features.
262 
263 Format: ASCII text
264 
265 \b Note: The contents of this file may also be specified as command line flags.
266 
267 This file contains all the inputs required for the PETSc time integrators. See PETSc documentation (https://petsc.org/release/docs/)
268 for all the inputs that PETSc needs, or <B>see the PETSc examples</B> for the inputs relevant to HyPar. In addition, following
269 are the HyPar-specific inputs (they are all optional, if not specified, default values are used):
270 + <B>-use-petscts</B>: If this flag is specified, PETSc time integration is used. If not specified, native time integration is used
271  (default).
272 + <B>-jfnk_epsilon \<value\></B>: specify \f$\epsilon\f$ parameter for the directional-derivative-based approximation of the Jacobian
273  (relevant only for implicit and IMEX time integration) (default: \f$10^{-6}\f$).
274 + <B>-with_pc</B>: If this flag is specified, a preconditioning matrix will be assembled for use with the preconditioners available
275  in PETSc. If not specified, no preconditioning will be used (default).
276 
277 In addition, if an IMEX time integrator (TSARKIMEX - https://petsc.org/release/docs/manualpages/TS/TSARKIMEX.html)
278 is being used, the following terms specify how the hyperbolic, parabolic,
279 and source terms are integrated in time (explicitly or implicitly):
280 + <B>-hyperbolic_explicit</B>: treat hyperbolic term explicitly (\b default).
281 + <B>-parabolic_explicit</B>: treat parabolic term explicitly.
282 + <B>-source_explicit</B>: treat the source term explicitly.
283 + <B>-hyperbolic_implicit</B>: treat hyperbolic term implicitly.
284 + <B>-parabolic_implicit</B>: treat parabolic term implicitly (\b default).
285 + <B>-source_implicit</B>: treat the source term implicitly (\b default).
